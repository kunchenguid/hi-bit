# Game assets

Use this skill when a 2D creation needs real game art that moves or needs a see-through background.
This includes characters, creatures, players, enemies, NPCs, projectiles, pickups, icons, and effects.
For a single flat picture that does not move and does not need transparency (a background, a logo, a photo-like illustration), plain `generate_image` is enough and you do not need this skill.
For 3D games, use plain `generate_image` textures on primitives from the `create-3d-game` skill instead of sprite sheets.

## Hard rule (read this first)

The sprite art MUST be generated by `generate_image`, then cleaned up by `process_sprite_sheet`.
Do NOT draw the sprite yourself in code.
That means: no PIL / Pillow, no Python `ImageDraw`, no drawing shapes with canvas, SVG, or CSS, no procedural pixel art, no writing a script that paints rectangles and circles to fake a character.
Even if Python and PIL are available, do not use them to make the art - they produce blocky, generic shapes, not the friendly character the kid asked for.
Code may only display the finished frames (in the game) or run `process_sprite_sheet`; it may never originate the art.
If you catch yourself about to write an image-drawing script, stop and call `generate_image` instead.

## The idea: draw on magenta, then process

Image models are great at drawing but bad at making clean transparent grids.
So we split the work in two:

1. Ask `generate_image` to draw the frames on a solid flat magenta `#FF00FF` background, laid out in an even grid.
   If your job lists reference picture ids, pass them in `reference_paths` so the sprite matches that character, colors, or style.
   Those ids may name pictures the builder shared, pictures found with `search_image`, or art made by `generate_image` in this or another creation.
2. Run the `process_sprite_sheet` tool, which keys out the magenta, slices the frames, lines them up, checks quality, and writes a transparent sheet, an animated GIF, and a `sprite-meta.json`.

Then wire the result into the creation with the tiny renderer in `references/sprite-renderer.js` so the art actually animates in the live preview.

Never hand-draw sprite frames with canvas, CSS, SVG, or shapes in code.
Real sprite art must come from `generate_image`; code only displays or processes it.

## Step 1: plan the sheet

Pick the smallest sheet that does the job.
One sheet should hold one action (one walk cycle, or one idle, or one attack), not several different actions mixed together.

Frame count to grid:

- 4 frames -> `2x2`
- 6 frames -> `2x3`
- 8 frames -> `2x4`
- 9 frames -> `3x3`
- 16 frames -> `4x4`

Do not use a single row like `1x4` for a character, creature, player, enemy, or anything with a body that must stay centered.
Single rows drift sideways and crop unevenly. Use a multi-row grid instead.
A `4x4` is fine when it is a four-direction walk (row 1 down, row 2 left, row 3 right, row 4 up).

## Step 2: write the prompt

Keep the art friendly and age-appropriate for kids 7-12 by default: bright, rounded, cheerful, simple, readable shapes.
Avoid gore, blood, realistic weapons, scary or violent imagery unless the kid clearly asked for something spooky-but-still-kid-safe (like a cartoon ghost).
Match the kid's theme and any art already in their creation.
If you have a reference picture id as art direction, pass it to `generate_image.reference_paths` instead of only describing it in words.

Every sprite-sheet prompt must say all of these:

- Solid flat magenta `#FF00FF` background, no gradients, no shadow under the subject.
- The exact grid (for example "an even 2x2 grid, 4 equal cells").
- No borders or lines between cells; cells are joined by magenta only.
- The same character with the same colors and the same size in every cell.
- The whole subject fits inside its cell with magenta margin on all sides; nothing crosses a cell edge.
- No text, no labels, no numbers, no UI, no speech bubbles.

Describe the motion cell by cell (for a walk: feet together, left foot forward, feet together, right foot forward).
See `references/prompt-rules.md` for ready-made patterns.

Save the raw sheet somewhere like `assets/raw/<name>.png`.

## Step 3: process the sheet

Call `process_sprite_sheet` with the raw file, an output folder, and the grid:

- `scaleStrategy: "preserve"` with `align: "feet"` for characters, creatures, players, enemies, and anything that stands on the ground. This keeps the drawn size and only lines up the feet, so capes, weapons, and wide poses are not squashed.
- `scaleStrategy: "fit"` for projectiles, icons, pickups, and effects, where filling the cell evenly matters more.
- Use the SAME strategy for every action sheet of one character. Mixing makes the body jump in size between actions.
- Pass `rejectEdgeTouch: true` when you want the tool to fail (so you can regenerate) if any frame touches a cell edge.

The tool writes `raw-sheet-clean.png`, `sheet-transparent.png`, the frame PNGs, `animation.gif`, and `sprite-meta.json` into the output folder.

## Step 4: check quality

Read the tool's reply and the `sprite-meta.json`.

- If `edgeTouchFrames` is not empty, a subject is jammed against a cell edge. Regenerate the raw sheet with more margin, or reprocess.
- If `scaleDrift` is high (more than about 0.2) on a character, the body changes size too much between frames. Regenerate the raw sheet asking for the same size in every cell.
- Open `animation.gif` mentally: does it read as one smooth, consistent animation?

## Step 5: wire it into the creation

Copy `references/sprite-renderer.js` into the creation (for example into the same assets folder), and use it:

```html
<script src="sprite-renderer.js"></script>
<script>
  let hero;
  HiBitSprite.load("assets/sprites/hero/sprite-meta.json").then((s) => { hero = s; });

  // in your animation loop, with ctx = canvas.getContext("2d"):
  function loop(deltaMs) {
    ctx.clearRect(0, 0, canvas.width, canvas.height);
    if (hero) {
      hero.update(deltaMs);   // advance the animation
      hero.draw(ctx, x, y);   // draw the current frame at x, y
    }
    requestAnimationFrame(/* ... */);
  }
</script>
```

`load` returns a sprite; call `update(deltaMs)` once per loop to advance it and `draw(ctx, x, y, scale)` to paint the current frame.
You do not have to use the renderer if the creation already has its own sprite code, but the sheet plus `sprite-meta.json` is all it needs: `frameWidth`, `frameHeight`, `columns`, `frameCount`, and `fps`.
